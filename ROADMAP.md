# PolicyCodex Roadmap

*Last updated 2026-06-11. Living document. Themes evolve as real-world install signal arrives.*

## How to read this

PolicyCodex follows a maintainer-mode open-source model. Dates appear only after we can promise them. Earlier phases describe themes and direction, not commitments. Anything labeled "vision" or "horizons" is genuinely uncertain and depends on partner engagement, real-world install signal, and the time of the maintainer team.

If you are evaluating PolicyCodex for your diocese or for a partnership, the most useful sections are **Architectural foundation** (what is built) and **In rebuild** (what is in progress before v0.1 ships). Everything past that is forward-looking and may change.

## Architectural foundation (built, survives the 2026-06-11 pivot)

The pieces below shipped pre-pivot and carry forward into v0.1 unchanged:

- Local-folder ingest with incremental re-runs and per-file content-hash diffing.
- AI inventory pass grounded in the diocese's own retention policy as a foundational bundle (one document plays two roles: inventory item and source-of-truth configuration).
- Per-diocese private Git repo with PR-as-gate workflow. Drafted = open PR, Reviewed = approved, Published = merged.
- Static handbook generated by Astro and published via GitHub Actions to a custom subdomain.
- Modern admin UI built with Tailwind CSS, DaisyUI, HTMX over Django templates, Inter typeface, brand color via DaisyUI CSS variables.
- Single-tenant self-hosted VM per diocese. One-command Docker Compose install. State persisted in a named volume.
- AGPL-3.0 license. Maintainer-mode services revenue (setup, support, customization). No licensing fees.

**Install zero:** Diocese of Pensacola-Tallahassee, live at `https://handbook.ptdiocese.org/`.

**Design reviewers:** Archdiocese of Los Angeles (David Schmitt, IT Director) and Archdiocese of Baltimore (Marcus Madsen, Director of IT).

## In rebuild (v0.1 ship pending the Settings-page architecture, post-2026-06-11 pivot)

The DISC-readiness sprint built a seven-screen onboarding wizard. The live walkthrough on 2026-06-11 surfaced architectural fault lines between the new credential store and the legacy settings-driven plumbing. Each fix revealed the next broken consumer. The team paused and called for a rethink.

The replacement architecture, locked the same evening:

- Seeded admin (`admin`/`admin1234`) forced to change password on first login.
- All configuration lives in a `/settings/` page with five panels: GitHub App, AI provider, Policy repository, Diocese configuration, Users-and-roles, plus a Reset panel.
- Inventory becomes a top-level `/inventory/` page with an always-present drop bucket.
- GitHub App provisioning automates via the GitHub App manifest flow with Installation ID auto-detect and a repo dropdown. The admin clicks three buttons instead of pasting an App ID, an Installation ID, and a PEM.

Six-phase rebuild plan, 30 tasks, roughly one week of focused work with parallel subagent dispatch. v0.1 ships when the rebuild lands. The architectural foundation above carries forward unchanged.

## Transition (between v0.1 ship and v0.2 commitment)

What the maintainer team does in the months between the Settings-page rebuild landing and v0.2 scope commitment. This phase exists deliberately. We do not commit v0.2 scope speculatively.

**What the maintainer team focuses on during the transition:**

- Closing out the v0.1 codebase cleanly. Snapshot the artifacts so the next cycle opens with a clean working state.
- Tracking install reports from dioceses that try PolicyCodex post-ship. Friction, requested features, and partnership conversations all feed v0.2 prioritization.
- Deepening engagement with the reference design partners. The Archdiocese of Los Angeles is the strongest install-N candidate and is the conversation focus.
- Quiet architectural prep work that does not commit feature scope. Adding an optional document-type field to policy frontmatter, extending the AI inventory prompt to suggest type alongside category, and baseline-testing classification accuracy against a labeled set. Substrate work, not surface.
- Active outreach to diocesan IT directors. A brief listserv email post-ship, then personalized one-to-one follow-ups.
- Resisting the urge to build v0.2 features speculatively. The right question is "what do real adopters need," not "what do we want to build."

The transition phase has no hard end date. Target close: end of August 2026. Three trigger criteria determine what happens next:

**Strong signal (v0.2 commits immediately):** one paying engagement booked, or one contributing partner with a substantive pull request, or one diocese running PolicyCodex in production for at least 30 days.

**Medium signal (v0.2 commits after a check-in):** one or two diocesan IT directors with installs in progress, plus documented feedback shaping what v0.2 should prioritize. This is the most likely scenario.

**Weak or no signal:** v0.2 holds. Transition either extends another month with more focused outreach, or the maintainer team steps back and v0.1 stays as a public artifact. The maintainer-mode business depends on real adopters. No adopters means no business to build for, and that is a legitimate outcome.

## Active Design (v0.2, scope pending the transition signal)

We are deliberately holding v0.2 scope open until the transition phase produces real signal. The themes below describe the candidate directions. Specific features become committed work only when the people who use PolicyCodex have told us what they need.

**Themes under consideration:**

**Generic-ship hardening.** Multi-diocese installs reveal edge cases in the Settings configuration flow, the foundational bundle workflow, or the handbook build path. First-priority work because it unblocks every adopter that follows. Wins compound across the installed base.

**Document type substrate.** Internal data-model evolution to recognize the full Policy / Standard / Guideline / Process / SOP stack. Frontmatter carries `document_type`. The AI inventory pass learns to suggest type alongside category and retention. UI vocabulary stays "policy" externally for now. Builds on the prep work started in the transition phase. Low risk, non-breaking, ships even without strong adopter signal.

**Diagnostic reclassification report.** Admin-only Django view that shows the diocesan IT director per-document type labels with confidence scores. The IT director validates accuracy privately, curates obviously-wrong labels, and produces a curated report they can show executive leadership as ROI substantiation. "Before PolicyCodex, every document was on the committee's agenda. After PolicyCodex, only the 11 actual policies reach the board." This is NOT a customer-facing differentiation feature. It is the IT director's internal selling tool. The CFO does not see it; the bishop does not see it; it lives in admin Settings.

**Type-aware gate routing.** Extension of the v0.1 three-gate flow into the full Seven-Gate workflow (PRD P2.2). Gates flex by document type rather than running every document through every gate. SOPs ship through the SME and publication gates. Policies still reach the board. Saves committee time on operational documents that should never have reached committee.

**Connector framework.** Source-of-truth integrations beyond local-folder ingest: SharePoint, OneDrive, Google Drive, Box, Dropbox, plus the Git-inbox pattern via GitHub Actions for dioceses that want event-driven ingest. Priority order set by what actual adopters request, not by a guess made today.

**Document owner upload flow.** UI surface for non-technical users to add a single new policy without waiting for the next bulk inventory pass. Includes semantic overlap detection against existing policies via an embedding index, and a three-way resolution choice (merge into existing, replace and deprecate the old, or create as parallel).

**AI provider cost controls.** Optional monthly spend ceiling at the provider abstraction layer, test-connection button in the Settings AI provider panel, and per-policy audit logging extended with token counts and timestamps.

**Settings-managed handbook publishing.** Closes the v0.1 manual DNS and GitHub Pages configuration gap by collecting the subdomain in the Settings Policy-repository panel, committing a CNAME file automatically, and where the GitHub App scope permits, setting the GitHub Pages custom domain via API.

**What "active design" means in practice:** brainstorms, mockups, architectural notes, and PRD P2 entries exist for these themes. Sprint-level tickets get filed only when we commit to ship.

## Vision (v0.3 and beyond)

Genuinely aspirational. Depends on partner engagement maturing and on the v0.2 substrate landing first. Naming these themes here gives prospective contributors a sense of where the project goes if it succeeds.

**Themes worth naming:**

**Public reclassification claim (the marketing moment, backed by real install data).** Once a diocese has run the v0.2 diagnostic report and curated their numbers, PolicyCodex earns the right to make the marketing claim publicly: "We sort your library into the five formal document types and tell you how much committee time you save." This is not a v0.3 product feature, it is a v0.3 marketing milestone backed by a quoted install report. The product feature shipped in v0.2 (the diagnostic tool). The public differentiation lands when the install evidence supports it. Holding the claim until the evidence exists protects credibility.

**Cross-document hierarchy detection.** "This SOP implements this Process which satisfies this Policy." Allows the handbook to render relationships, allows compliance review to walk down the tree, and allows audit prep to identify which SOPs trace back to which regulatory requirement. Requires more sophisticated AI prompting than the flat-label diagnostic in v0.2. Layer on top once the flat labels are trustworthy.

**Type-aware authoring.** A CFO clicks "Create new document" and the first question is the document type. Each type picks a different template, a different gate path, and a different review surface. Builds on the conversational AI authoring foundation introduced in v0.2.

**Change-authority enforcement.** The gate matrix says some document types can be revised by their owning team without re-running gates. Build that into the PR flow. Reduce committee load on routine SOP and guideline updates.

**Q&A chatbot over the published handbook (RAG).** Parish staff member asks "how long do we keep baptismal records" and gets the relevant policy plus the retention rule. Same embedding index that powers overlap detection in v0.2.

**Multilingual handbook.** AI-translated drafts, human-reviewed, ships through the same PR gate. Spanish first because Hispanic ministry is a major priority for most US dioceses.

## Strategic Horizons (v0.4+ and uncertain)

Where the maintainer-mode business compounds if the platform plays come together. Real possibility but no commitment. These are conversations to have with partners, not promises to ship.

**Possible horizons:**

- **AI generation of standards from policies.** "You have a password policy. Here are five standards you should consider as implementation specifics." The diocese reviews and accepts what fits, ignores what does not.
- **AI generation of SOPs from processes.** Massive content-generation lift for diocesan operations. Translates the documented process into department-level keystrokes.
- **Multi-document compliance auditing.** "Your password standard requires 14 characters. Your IT SOPs reference 8 characters in three places. Fix recommendations attached." Cross-document consistency checking at scale.
- **Regulatory change watching.** PolicyCodex notices USCCB updated the Charter for the Protection of Children, identifies the affected policies in the diocese, drafts the updates.
- **Cross-diocese benchmarking** (carefully, with explicit consent). "Dioceses similar to yours typically have three standards under their financial-controls policy. You have zero." Privacy-preserving and opt-in.
- **Self-hosted serving variant.** Caddy or Nginx reverse proxy on the diocese's own VM for dioceses that prefer not to use GitHub Pages.

## Permanent non-goals

Some things will not be built no matter how the roadmap evolves. Naming them here saves contributors and prospective partners from proposing work the project will refuse.

- **Multi-tenant SaaS hosting.** Each diocese runs its own VM. The maintainer-mode pitch depends on operational independence.
- **Auto-publishing of AI drafts.** Every published document carries the diocese's institutional weight. The audit-trail guarantee in PRD G3 is a hard line. Every published version has a named human approver.
- **Multi-LLM consensus reviews.** One model plus a sharp rubric plus a human approver wins. Multi-LLM consensus is overbuilt for the problem.
- **Commercial component libraries.** The AGPL license forbids redistribution of Tailwind UI, MUI Pro, Ant Design Pro, DevExpress, Telerik, Kendo UI, and similar paid components. The project stays on permissive-license dependencies for as long as it remains AGPL.
